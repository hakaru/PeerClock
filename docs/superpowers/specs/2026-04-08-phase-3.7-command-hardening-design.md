# Phase 3.7 — CommandRouter Hardening — Design

**Date:** 2026-04-08
**Status:** Draft
**Phase:** 3.7 (between Phase 3.6 and 1Take MVP integration)

## Background

Phase 3.6 の実装中に 2 つの問題が露呈した:

1. **Problem 2 (preexisting race)**: `CommandRouter.syncMessages` は単一 AsyncStream だが、`PeerClock.syncResponder` と `NTPSyncEngine.collectMeasurements` の両方が同時に `for await` で消費する。AsyncStream は FIFO で 1 要素 1 consumer のため、PONG が syncResponder に吸われて listener に届かないケースが発生し follower の同期が失敗する。

2. **四者討論の結論**: 1Take MVP 統合では `APP_COMMAND` の重複排除・順序保証が必須だが、現状の `CommandRouter` は commandID / logicalVersion を持たない。1Take 側 Adapter が「リクエスト→確定」パターンを安全に実装するためには、Router 単一点での identity 管理が前提条件となる。

Phase 3.7 はこの 2 つを同時に修正する (両方 1Take MVP の前提条件なので 1 plan に統合)。

## Goals

- `syncMessages` を `syncRequests` (PING) と `syncResponses` (PONG) に分離し、syncResponder と sync listener が独立ストリームを消費するようにする
- `APP_COMMAND` に `commandID: UUID` と `logicalVersion: UInt64` を wire に追加
- `CommandRouter.send` / `broadcast` で自動付与
- 受信側で (sender, commandID) 重複排除と (sender, version) 前進保証
- メモリ上限: sender あたり直近 1000 件 LRU
- `protocolVersion = 2` に bump (v1 wire は decode reject)

## Non-Goals

- sender 間の logical clock / causal ordering (Phase 4 の合議ベース領域)
- リクエスト→確定パターンの API 化 (1Take 側に閉じ込める方針、Phase 3 ラウンド 3 で確定)
- 持続化された command history (ring buffer の LRU のみ)
- v1 wire との後方互換 (1Take 未統合のため不要)

## Architecture

```
CommandRouter (拡張)
  ├─ incomingCommands: AsyncStream<(PeerID, Command)>   [既存、受信済み識別子で重複排除後]
  ├─ syncRequests: AsyncStream<(PeerID, Message)>       [新規、PING のみ]
  ├─ syncResponses: AsyncStream<(PeerID, Message)>      [新規、PONG のみ]
  ├─ (削除) syncMessages
  │
  ├─ State (actor 内部):
  │   nextLogicalVersion: UInt64 = 1
  │   seenCommands: [PeerID: Set<UUID>]
  │   seenOrder: [PeerID: [UUID]] (FIFO 最大 1000)
  │   highestVersionPerSender: [PeerID: UInt64]
  │
  ├─ send(_ command, to:) — commandID/version を自動付与して wire encode
  ├─ broadcast(_ command)  — 同上
  ├─ ingestAppCommand — 重複排除 + 前進チェック後に incomingCommands に yield
  └─ forgetPeer(PeerID)   — disconnect 時に state を消去

Wire Protocol v2
  APP_COMMAND payload = [16B commandID UUID] [8B logicalVersion BE UInt64] [既存の type + data]
  Configuration.protocolVersion = 2
```

## Components

### Subfeature A: Stream split

**`CommandRouter`** の既存 `syncMessages` を削除し、PING/PONG 用に 2 ストリームを公開:

```swift
public let syncRequests: AsyncStream<(PeerID, Message)>
public let syncResponses: AsyncStream<(PeerID, Message)>

private let syncRequestsCont: AsyncStream<(PeerID, Message)>.Continuation
private let syncResponsesCont: AsyncStream<(PeerID, Message)>.Continuation
```

Router の受信 dispatch でメッセージ種別により振り分け:
```swift
switch message {
case .ping:
    syncRequestsCont.yield((sender, message))
case .pong:
    syncResponsesCont.yield((sender, message))
// ... その他 ...
}
```

**`PeerClock.startSyncResponder`**: `commandRouter.syncMessages` → `commandRouter.syncRequests`
**`NTPSyncEngine`**: init 引数の `syncMessageStream` を `syncResponseStream` に rename し、PeerClock から `commandRouter.syncResponses` を渡す

### Subfeature B: Command identity

**Wire Format (Message.swift / MessageCodec.swift)**:

`Message` enum の `.appCommand` case に `commandID: UUID` と `logicalVersion: UInt64` を追加:

```swift
case appCommand(commandID: UUID, logicalVersion: UInt64, senderID: PeerID, command: Command)
```

`MessageCodec.encode` の APP_COMMAND 分岐で 16B UUID + 8B BE UInt64 を wire に書き込む。`decode` も対応。

`Configuration.protocolVersion: UInt16 = 2`。HELLO / negotiation で v1 を受け取ったら mismatch エラーで reject。

**CommandRouter.send / broadcast**:

```swift
public actor CommandRouter {
    private var nextLogicalVersion: UInt64 = 1

    public func send(_ command: Command, to peer: PeerID) async throws {
        let id = UUID()
        let version = nextLogicalVersion
        nextLogicalVersion += 1
        let msg = Message.appCommand(
            commandID: id,
            logicalVersion: version,
            senderID: localPeerID,
            command: command
        )
        try await transport.sendReliable(MessageCodec.encode(msg), to: peer)
    }

    public func broadcast(_ command: Command) async throws {
        let id = UUID()
        let version = nextLogicalVersion
        nextLogicalVersion += 1
        let msg = Message.appCommand(
            commandID: id,
            logicalVersion: version,
            senderID: localPeerID,
            command: command
        )
        try await transport.broadcastReliable(MessageCodec.encode(msg))
    }
}
```

**受信側 dispatch** (ingestAppCommand):

```swift
private var seenCommands: [PeerID: Set<UUID>] = [:]
private var seenOrder: [PeerID: [UUID]] = [:]
private var highestVersionPerSender: [PeerID: UInt64] = [:]
private let maxSeenPerSender = 1000

private func ingestAppCommand(
    sender: PeerID,
    commandID: UUID,
    logicalVersion: UInt64,
    command: Command
) {
    // 1. 重複排除
    if seenCommands[sender, default: []].contains(commandID) {
        return
    }

    // 2. Version 前進チェック
    if let highest = highestVersionPerSender[sender], logicalVersion < highest {
        return
    }

    // 3. 受理: state 更新
    var seen = seenCommands[sender, default: []]
    var order = seenOrder[sender, default: []]
    seen.insert(commandID)
    order.append(commandID)
    // LRU eviction
    if order.count > maxSeenPerSender {
        let evict = order.removeFirst()
        seen.remove(evict)
    }
    seenCommands[sender] = seen
    seenOrder[sender] = order
    highestVersionPerSender[sender] = max(
        highestVersionPerSender[sender] ?? 0, logicalVersion
    )

    // 4. yield
    incomingCommandsCont.yield((sender, command))
}
```

**ピア切断時の state 消去**:

`CommandRouter.forgetPeer(_ peer: PeerID)` を追加し、PeerClock の heartbeat disconnect ハンドラから呼ぶ:

```swift
public func forgetPeer(_ peer: PeerID) {
    seenCommands[peer] = nil
    seenOrder[peer] = nil
    highestVersionPerSender[peer] = nil
}
```

再接続時は version 1 から再開される (新規 peer 扱い)。

## Data Flow

```
[Sender] CommandRouter.send(command, to: peer)
            ↓
         UUID + nextLogicalVersion++
            ↓
         Message.appCommand(commandID, version, senderID=self, command)
            ↓
         MessageCodec.encode (v2: [UUID 16B][version 8B][type][data])
            ↓
         transport.sendReliable

[Receiver] transport inbound → router dispatch
            ↓
         switch message
            ├─ .ping → syncRequestsCont.yield
            ├─ .pong → syncResponsesCont.yield
            └─ .appCommand → ingestAppCommand
                  ├─ dedup check (seenCommands)
                  ├─ version forward check
                  ├─ update state (seen, order, highest)
                  └─ incomingCommandsCont.yield((sender, command))

[Peer disconnect] HeartbeatMonitor.disconnected → PeerClock → router.forgetPeer(peer)
```

## Testing

### CommandRouterIdentityTests
1. **自動付与**: `router.send(cmd, to: peer)` → MockTransport に載った raw bytes を decode すると commandID と logicalVersion が載っている
2. **重複排除**: 同一 (sender, commandID) を 2 回 ingest → incomingCommands に 1 度だけ流れる
3. **Version 前進**: 同一 sender から v=3, v=5, v=4 を順に ingest → v=3, v=5 が流れ v=4 は drop
4. **独立 sender**: 異なる sender の同じ commandID は衝突しない (両方流れる)
5. **LRU eviction**: `maxSeenPerSender` を 3 に下げたテスト config で、4 番目を insert すると最古が evict される
6. **forgetPeer**: peer A を forget → A の seen/highest が消える → 同じ commandID を再 ingest すると通過する

### CommandRouterStreamSplitTests
7. **PING は syncRequests のみ**: PING 受信後に syncRequests が yield、syncResponses には届かない
8. **PONG は syncResponses のみ**: 同上で逆
9. **他のメッセージは不変**: APP_COMMAND/HEARTBEAT が syncRequests/syncResponses にも流入しない

### WireCodecTests 追加
10. **v2 appCommand round-trip**: encode → decode で commandID と logicalVersion が保たれる
11. **v1 appCommand decode**: 旧形式を decode すると failing / mismatch (テスト bytes を手書きで構築)

### 既存テスト migration
- 既存 `commandRouter.syncMessages` 参照箇所 (PeerClock.swift, NTPSyncEngine.swift, NTPSyncEngineTests.swift) を `syncRequests` / `syncResponses` に置換
- `Message.appCommand(...)` を直接構築している箇所 (テスト含む) を新しい signature に合わせる

## Error Handling

- Wire v1 受信: `MessageCodec.decode` が `.corrupt` or `.unsupportedVersion` を返す → transport 層でログ + drop
- LRU overflow: 防衛的に `order.removeFirst()` + `seen.remove(evict)` で無制限成長を防ぐ
- `seenCommands` の actor 境界: `CommandRouter` は actor なので並行 ingest は自動シリアライズ
- `forgetPeer` を忘れた場合: メモリリークはするが LRU で上限あり。致命的ではない

## Migration

1. **Phase 3.7 本体**: CommandRouter + Wire codec + PeerClock/NTPSyncEngine の stream 参照変更
2. **既存テスト**: `Message.appCommand(...)` 直接構築と `syncMessages` 参照を置換
3. **Protocol version 2**: 1Take 未統合なので本番影響ゼロ

## Open Questions

なし。

## Risks

- **`Message.appCommand` の case 変更**: enum の associated values 変更は既存 switch 全てに影響。grep で漏れを洗い出す
- **forgetPeer の呼び出し漏れ**: heartbeat disconnect フックに追加する必要あり。Phase 3a で既存の disconnect ハンドラがどこにあるか確認
- **NTPSyncEngine の syncMessageStream 引数名変更**: existing call sites (NTPSyncEngineTests 含む) を全て更新
- **LRU FIFO の O(n) removeFirst**: Array の先頭削除は O(n)。maxSeenPerSender=1000 では実用上問題なし。将来必要なら Deque に差し替え
