# Phase 3.6 — Sync Guard & Schedule API Hardening — Design

**Date:** 2026-04-08
**Status:** Draft
**Phase:** 3.6 (between Phase 3.5 and 1Take MVP integration)

## Background

Phase 2b で `PeerClock.schedule(atSyncedTime:)` API を実装したが、1Take への組み込みを四者討論 (Claude / Codex / Gemini / ユーザー) で検証した結果、現状の API では予約録音に致命的な欠陥がある:

1. **同期前の schedule が無条件受理される** — `now` は未同期でも offset 0 で進み、誤った時刻でトリガされる
2. **同期信頼度を考慮しない** — quality が低くても受理
3. **過去時刻 schedule の契約が未定義** — 現状 `.missed` でも action 実行が既定
4. **DriftMonitor jump 時に pending event が再スケジュールされない** — `EventScheduler.handleJump` は警告を出すだけで、Sleeper はそのまま古い時間軸で寝続ける

これらは Phase 4 の合議ベース同期に進む前提条件として、また 1Take MVP 統合に進む前提条件として、すべて MVP 前必須。

## Goals

- スケジュール時に同期状態と信頼度を検証し、不適切な要求を early reject
- 過去時刻スケジュールの挙動を明示的に制御
- DriftMonitor jump 後に pending event を新時間軸で再スケジュール
- API は既存 1 メソッドを breaking change で増強 (引数追加 + throws)。未本番なので影響少

## Non-Goals

- `willExecute(leadTime:)` プリトリガコールバック (1Take 側で「schedule 受信時 prepare」で代替可)
- 同時複数 schedule の調停 (順序保証不要)
- 自動再ガード解除 (`isSynchronized` 公開で自然に実装可)
- リクエスト→確定パターンの API 化 (Phase 4 領域、1Take 側で実装)
- サンプリングレート個体差補正 (Phase 4)

## Architecture

```
PeerClock (Facade)
  ├─ currentSync: SyncSnapshot      [新規 read API]
  ├─ schedule(atSyncedTime:lateTolerance:_:) async throws -> ScheduledEventHandle
  │       ├─ (1) currentSync.isSynchronized 検査 → notSynchronized
  │       ├─ (2) quality.confidence < Configuration.minSyncQuality → qualityBelowThreshold
  │       ├─ (3) atSyncedTime < now で lateNs > toleranceNs → deadlineExceeded
  │       └─ EventScheduler.schedule に委譲
  └─ EventScheduler (actor)
       └─ handleJump で全 pending event を cancel + 再スケジュール  [新規実装]
```

## Components

### SyncSnapshot

```swift
public struct SyncSnapshot: Sendable {
    /// 直近の同期状態 (.synced のみが「同期済み」)
    public let state: SyncState
    /// 現在のオフセット (秒、未同期時は 0)
    public let offset: TimeInterval
    /// 直近の品質情報 (未同期時は nil)
    public let quality: SyncQuality?
    /// 直近に .synced へ遷移した時刻 (CLOCK_MONOTONIC ns、未同期時は nil)
    public let lastSyncedAt: UInt64?
    /// このスナップショットを取得した時刻 (CLOCK_MONOTONIC ns)
    public let capturedAt: UInt64

    /// 同期済み かつ Configuration.syncStaleAfter 以内に最終同期している
    public var isSynchronized: Bool { ... }
}
```

`PeerClock.currentSync: SyncSnapshot` は lock 配下で即時計算する read プロパティ。

### Configuration 追加

```swift
/// schedule() がガードする最小同期信頼度 (0.0..1.0)。デフォルト 0.5。
public let minSyncQuality: Double

/// SyncSnapshot.isSynchronized が true となる最終同期からの最大経過時間。
/// デフォルト 90 秒 (Phase 3.5 のバックオフ最大 30s × 2 + マージン)。
public let syncStaleAfter: Duration
```

### PeerClockError

```swift
public enum PeerClockError: Error, Sendable {
    case notSynchronized
    case qualityBelowThreshold(quality: Double, threshold: Double)
    case deadlineExceeded(lateBy: Duration, tolerance: Duration)
}
```

### schedule() 強化版

```swift
public func schedule(
    atSyncedTime: UInt64,
    lateTolerance: Duration = .zero,
    _ action: @escaping @Sendable () async -> Void
) async throws -> ScheduledEventHandle
```

ガード順:
1. `currentSync.isSynchronized` を検査
2. `quality.confidence` を `Configuration.minSyncQuality` と比較
3. `atSyncedTime < now` の場合、`(now - atSyncedTime)` を `lateTolerance` と比較。
   `lateTolerance` 内なら EventScheduler に渡して即時発火扱い、超過なら throw。
4. `EventScheduler.schedule(atSyncedTime:_:)` に委譲

### EventScheduler.handleJump 修正

現状 `driftWarning` を発行するだけ。修正後:

1. 全 pending event の Task を cancel
2. 各 event を新しい `now()` で delay 再計算
3. delay <= 0 のものは即時発火、それ以外は新 Task で再 sleep
4. 既存の `driftWarning` 通知も維持 (subscriber 後方互換)

実装時に既存の `EventScheduler` 内部構造 (Event 構造体、pendingEvents の field 名) を読んで signature を合わせる。

## Data Flow

```
Caller ──► PeerClock.schedule(atSyncedTime, lateTolerance, action)
              │
              ├─ snapshot = currentSync
              ├─ guard isSynchronized       ─► throw notSynchronized
              ├─ guard quality              ─► throw qualityBelowThreshold
              ├─ check lateness             ─► throw deadlineExceeded
              └─ scheduler.schedule(...)    ─► ScheduledEventHandle

DriftMonitor.jumps ──► PeerClock ──► EventScheduler.handleJump(old, new)
                                          │
                                          ├─ cancel all pending Tasks
                                          ├─ for each event:
                                          │   delay = atSyncedTime - now()
                                          │   if delay <= 0: fire now
                                          │   else: new sleep Task
                                          └─ yield .driftWarning(delta)
```

## Testing

### SyncSnapshotTests
- `isSynchronized` true: state=.synced + capturedAt - lastSyncedAt < staleAfter
- `isSynchronized` false (state): state=.discovering / .syncing / .idle
- `isSynchronized` false (鮮度): state=.synced だが経過時間 > staleAfter
- 境界値: 経過時間 == staleAfter ちょうど

### PeerClockSyncGuardTests
- 未同期で `schedule` → `notSynchronized` throw
- `Configuration.minSyncQuality = 0.9` で `quality.confidence = 0.5` → `qualityBelowThreshold` throw
- 同期後に `syncStaleAfter` 超過待機 → `notSynchronized` throw
- 過去時刻 + `lateTolerance: .zero` → `deadlineExceeded` throw
- 過去時刻だが `lateTolerance: .milliseconds(100)` 内 → 即時実行成功
- 正常 schedule → `now + 50ms` で発火

### EventSchedulerJumpTests
- 1 秒後発火予定 → 500ms 時点で +200ms ジャンプ → 約 700ms で発火 (新時間軸)
- 1 秒後発火予定 → 500ms 時点で -2 秒ジャンプ (過去化) → 即時発火

## Error Handling

- `Duration` → ナノ秒変換は `Int64` 範囲内 (秒 × 1e9 + as/1e9)。日単位を超える `lateTolerance` は実用上ありえない
- `currentSync` は actor を介さない synchronous read (lock 配下)。EventScheduler へ渡す段階で actor 境界を越える
- `handleJump` 中の race: EventScheduler は actor なので新規 schedule とシリアライズされる、問題なし

## Migration

- `schedule(atSyncedTime:_:)` (Phase 2b) は **breaking change**。
  既存呼び出し側は引数追加なし + `try` キーワード追加のみで対応可能。
- 1Take 未統合のため本番影響ゼロ
- PeerClockDemo / 既存テストは `try` キーワード追加と必要に応じた await/throws 伝播

## Open Questions

なし (四者討論で全論点解消済み)。

## Risks

- **`EventScheduler` 内部の signature 整合性**: 上記コードは概念図、実装時に既存 `pendingEvents` の field 名・Event 構造体・`fireEvent` のシグネチャを読んで合わせる必要あり
- **テストでのジャンプ再現**: handleJump を直接呼ぶか、DriftMonitor 経由でジャンプを起こすか。直接呼びの方がテスト独立性が高い
- **既存テストへの影響**: `schedule` を呼ぶ既存テストで `try await` 化が必要。grep で全箇所洗い出しが必要
