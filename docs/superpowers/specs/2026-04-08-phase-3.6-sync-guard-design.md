# Phase 3.6 — Sync Guard & Schedule API Hardening — Design

**Date:** 2026-04-08
**Status:** Draft (Codex review applied)
**Phase:** 3.6 (between Phase 3.5 and 1Take MVP integration)

## Background

Phase 2b で `PeerClock.schedule(atSyncedTime:)` API を実装したが、1Take への組み込みを四者討論 (Claude / Codex / Gemini / ユーザー) で検証した結果、現状の API では予約録音に致命的な欠陥がある:

1. **同期前の schedule が無条件受理される** — `now` は未同期でも offset 0 で進み、誤った時刻でトリガされる
2. **同期信頼度を考慮しない** — quality が低くても受理
3. **過去時刻 schedule の契約が未定義** — 現状 `forceMissed: true` で `tryFire` するが action は実行される
4. **DriftMonitor jump 時に pending event が再スケジュールされない** — `EventScheduler.handleJump` (Sources/PeerClock/EventScheduler/EventScheduler.swift:94) は `driftWarning` を yield するだけで pending Sleeper Task は古い時間軸で寝続ける

これらは Phase 4 の合議ベース同期に進む前提条件として、また 1Take MVP 統合に進む前提条件として、すべて MVP 前必須。

## Goals

- スケジュール時に同期状態と信頼度を検証し、不適切な要求を early reject
- 過去時刻スケジュールの挙動を明示的に制御
- DriftMonitor jump 後に pending event を新時間軸で再スケジュール
- API は既存 1 メソッドを breaking change で増強 (引数追加 + throws)。未本番なので影響少

## Non-Goals

- `willExecute(leadTime:)` プリトリガコールバック (1Take 側で「schedule 受信時 prepare」で代替可)
- 同時複数 schedule の調停 (順序保証不要)
- 自動再ガード解除 (`isSynchronized` 公開で自然に実装可)
- リクエスト→確定パターンの API 化 (Phase 4 領域、1Take 側で実装)
- サンプリングレート個体差補正 (Phase 4)
- async クロージャ対応 (現行 EventScheduler.Action は同期型のまま維持)

## Architecture

```
PeerClock (Facade)
  ├─ syncSnapshotState (lock 配下に保持される最新同期状態)
  │   updated by: syncStateForwardTask が eng.syncStateUpdates を消費するときに同時更新
  ├─ currentSync: SyncSnapshot      [新規 read API、lock 配下で snapshot を組み立てる]
  ├─ schedule(atSyncedTime:lateTolerance:_:) async throws -> ScheduledEventHandle
  │       ├─ (0) PeerClock 未起動 (eventScheduler == nil) → notStarted throw
  │       ├─ (1) currentSync.isSynchronized 検査 → notSynchronized
  │       ├─ (2) quality.confidence < Configuration.minSyncQuality → qualityBelowThreshold
  │       ├─ (3) atSyncedTime < now で lateNs > toleranceNs → deadlineExceeded
  │       └─ EventScheduler.schedule に委譲
  └─ EventScheduler (actor)
       └─ handleJump で全 pending event を cancel + 再スケジュール (新規実装)
```

## Components

### SyncSnapshot

```swift
public struct SyncSnapshot: Sendable {
    /// 直近の同期状態 (.synced のみが「同期済み」)
    public let state: SyncState
    /// 現在のオフセット (秒、未同期時は 0)
    public let offset: TimeInterval
    /// 直近の品質情報 (未同期時は nil)
    public let quality: SyncQuality?
    /// 直近に .synced へ遷移した時刻 (CLOCK_MONOTONIC ns、未同期時は nil)
    public let lastSyncedAt: UInt64?
    /// このスナップショットを取得した時刻 (CLOCK_MONOTONIC ns)
    public let capturedAt: UInt64
    /// 鮮度判定用の閾値 (Configuration.syncStaleAfter をナノ秒換算)
    private let staleAfterNs: UInt64

    /// 同期済み かつ Configuration.syncStaleAfter 以内に最終同期している
    public var isSynchronized: Bool {
        guard case .synced = state, let last = lastSyncedAt else { return false }
        return capturedAt >= last && capturedAt - last <= staleAfterNs
    }
}
```

### PeerClock — sync state holding strategy

`PeerClock` は既存の private `lock: NSLock` 配下に以下 3 フィールドを追加保持する:

```swift
private var lastSyncState: SyncState = .idle
private var lastSyncedAtNs: UInt64? = nil
```

更新タイミング: 既存の `syncStateForwardTask` (PeerClock.swift:227 付近) が `eng.syncStateUpdates` を消費するループに、lock 配下での状態保存処理を 1 ステップ追加する:

```swift
for await state in eng.syncStateUpdates {
    cont.yield(state)
    self.lock.withLock {
        self.lastSyncState = state
        if case .synced = state {
            self.lastSyncedAtNs = NTPSyncEngine.now()
        }
    }
    // ... 既存の registry.setStatus 群はそのまま継続 ...
}
```

`stop()` 時に `lastSyncState = .idle`, `lastSyncedAtNs = nil` にリセット (既存 stop 処理に追加)。

### PeerClock.currentSync read API

```swift
public var currentSync: SyncSnapshot {
    let captured = NTPSyncEngine.now()
    return lock.withLock {
        let q: SyncQuality? = {
            if case .synced(_, let quality) = self.lastSyncState { return quality }
            return nil
        }()
        let off: TimeInterval = {
            if case .synced(let offset, _) = self.lastSyncState { return offset }
            return 0.0
        }()
        return SyncSnapshot(
            state: self.lastSyncState,
            offset: off,
            quality: q,
            lastSyncedAt: self.lastSyncedAtNs,
            capturedAt: captured,
            staleAfterNs: self.configuration.syncStaleAfterNs
        )
    }
}
```

### PeerClockError

```swift
public enum PeerClockError: Error, Sendable {
    case notStarted
    case notSynchronized
    case qualityBelowThreshold(quality: Double, threshold: Double)
    case deadlineExceeded(lateBy: Duration, tolerance: Duration)
}
```

### schedule() 強化版

```swift
public func schedule(
    atSyncedTime: UInt64,
    lateTolerance: Duration = .zero,
    _ action: @escaping @Sendable () -> Void
) async throws -> ScheduledEventHandle
```

注: クロージャ型は **既存の `EventScheduler.Action` (`@Sendable () -> Void`) と一致させる**。async クロージャ対応は Phase 3.6 のスコープ外。

ガード順:
1. **未起動チェック**: `eventScheduler == nil` → `notStarted` throw
2. **同期チェック**: `currentSync.isSynchronized == false` → `notSynchronized` throw
3. **品質チェック**: `quality.confidence < minSyncQuality` → `qualityBelowThreshold` throw (`==` は通過)
4. **過去時刻チェック**: `atSyncedTime < now` の場合 `lateNs = now - atSyncedTime`。
   `lateNs > toleranceNs` (`==` は通過) なら `deadlineExceeded` throw。
   それ以外は `EventScheduler.schedule` に委譲 (内部で `tryFire(forceMissed: true)` 即時発火)。
5. `scheduler.schedule(atSyncedTime:_:)` に委譲して `ScheduledEventHandle` を返す

判定に使う `now` は `self.now` を **1 度だけ** 取得して同じ値を使う (snapshot の `capturedAt` とは別取得でよい)。

### EventScheduler.handleJump 修正

現状 (EventScheduler.swift:94-105):
```swift
public func handleJump(oldOffsetNs: Int64, newOffsetNs: Int64) {
    for (id, event) in events where event.state == .pending {
        logger.warning(...)
        eventContinuation.yield(.driftWarning(eventID: id, ...))
    }
}
```

修正後:
```swift
public func handleJump(oldOffsetNs: Int64, newOffsetNs: Int64) {
    let pending = events.filter { $0.value.state == .pending }
    for (id, event) in pending {
        logger.warning("Drift jump rescheduling event \(id.uuidString): old=\(oldOffsetNs) new=\(newOffsetNs)")
        eventContinuation.yield(.driftWarning(
            eventID: id,
            oldOffsetNs: oldOffsetNs,
            newOffsetNs: newOffsetNs
        ))

        // 1. 古い Sleeper Task を cancel (state は .pending のまま)
        event.task?.cancel()
        var updated = event
        updated.task = nil
        events[id] = updated

        // 2. 新しい now() で delay を再計算
        let delay = Int64(updated.atSyncedTime) - Int64(now())
        if delay <= 0 {
            // ジャンプで過去化 → 即時 fire (forceMissed)
            tryFire(id, forceMissed: true)
        } else {
            // 新 Task で再 sleep
            let waitNs = UInt64(delay)
            let task = Task {
                try? await self.sleeper.sleep(nanoseconds: waitNs)
                await self.tryFire(id, forceMissed: false)
            }
            updated.task = task
            events[id] = updated
        }
    }
}
```

**二重実行防止**: `tryFire` (EventScheduler.swift:120) は `event.state == .pending` の guard で 1 度しか fire しない既存実装 (cancel/fire race 対策と同じメカニズム)。再スケジュール時に新旧 Task の両方が `tryFire` を呼んでも、最初の 1 つだけが state を変えて action を実行する。

## Data Flow

```
Caller ──► PeerClock.schedule(atSyncedTime, lateTolerance, action)
              │
              ├─ guard eventScheduler != nil ─► throw notStarted
              ├─ snapshot = currentSync       (lock 配下で組み立て)
              ├─ guard isSynchronized         ─► throw notSynchronized
              ├─ guard quality                ─► throw qualityBelowThreshold
              ├─ check lateness               ─► throw deadlineExceeded
              └─ scheduler.schedule(...)      ─► ScheduledEventHandle

NTPSyncEngine.syncStateUpdates ──► PeerClock.syncStateForwardTask
                                       │
                                       ├─ yield to public stream
                                       ├─ lock { lastSyncState = state, lastSyncedAtNs = now() if synced }
                                       └─ registry.setStatus(...)

DriftMonitor.jumps ──► PeerClock.driftJumpTask ──► syncEngine.resetBackoff() (Phase 3.5)
                                                ──► EventScheduler.handleJump(old, new)
                                                       │
                                                       └─ for each pending event:
                                                            yield .driftWarning
                                                            cancel old Task
                                                            recompute delay with new now()
                                                            if delay <= 0: tryFire(forceMissed: true)
                                                            else: new sleep Task
```

## Testing

### SyncSnapshotTests
- `isSynchronized` true: state=.synced, capturedAt - lastSyncedAt < staleAfter
- `isSynchronized` 境界: capturedAt - lastSyncedAt == staleAfter ちょうど (true)
- `isSynchronized` false (state): state=.discovering / .syncing / .idle
- `isSynchronized` false (鮮度): 経過時間 > staleAfter
- `isSynchronized` false (capturedAt < lastSyncedAt の異常時計): 単調時計後退ガード

### PeerClockSyncGuardTests
- 未起動 PeerClock で `schedule` → `notStarted` throw
- 起動済み未同期で `schedule` → `notSynchronized` throw
- `Configuration.minSyncQuality = 0.9`、`quality.confidence = 0.5` → `qualityBelowThreshold` throw
- 境界: `quality.confidence == minSyncQuality` → 通過 (throw されない)
- 同期成功後に `syncStaleAfter` 超過待機 → `notSynchronized` throw
- 過去時刻 + `lateTolerance: .zero` → `deadlineExceeded` throw
- 境界: `lateBy == lateTolerance` → 通過 (throw されない)
- 過去時刻だが `lateTolerance: .milliseconds(100)` 内 → 即時実行成功 (handle 返却 + action 発火)
- 正常 schedule → `now + 50ms` で発火

### EventSchedulerJumpTests
- 1 イベント、1 秒後発火予定 → 500ms 時点で +200ms ジャンプ → 約 700ms (元の予定時刻 - 200ms 相当) で発火
- 1 イベント、1 秒後 → 500ms 時点で -2 秒ジャンプ (過去化) → 即時発火 (.missed 状態)
- **複数 pending イベント**: 3 イベントを 1s/2s/3s 後で schedule → 500ms 時点で +200ms ジャンプ → 全 3 件が新時間軸で正しく再スケジュールされる
- **二重実行防止**: ジャンプ発生時に新旧両方の Task が `tryFire` を呼ぶ可能性があるが、action は 1 度しか実行されない (count==1 アサート)

## Error Handling

- `Duration` → ナノ秒変換: `lateTolerance.components.seconds * 1_000_000_000 + attoseconds / 1_000_000_000`。秒は `Int64`、日単位を超える `lateTolerance` は実用上ありえない
- `currentSync` は actor を介さない synchronous read (lock 配下)。EventScheduler へ渡す段階で actor 境界を越える
- `handleJump` 中の race: EventScheduler は actor なので新規 schedule とシリアライズされる、問題なし
- 単調時計は理論上後退しないが防御コードとして `capturedAt >= lastSyncedAt` ガードを `isSynchronized` に入れる

## Migration

- `schedule(atSyncedTime:_:)` (Phase 2b) は **breaking change**: 引数追加 + `throws` 化 + `async` 化 (既に async)
  既存呼び出し側は引数追加なし + `try` キーワード追加のみで対応可能
- 1Take 未統合のため本番影響ゼロ
- PeerClockDemo / 既存テストで `schedule` を呼ぶ全箇所を grep で洗い出し、`try` を追加。
  `EventSchedulerTests` 系統 (actor 直接呼び出し) は `try` 不要 (EventScheduler.schedule は throw しない)。
- Configuration init 引数は default 値付き追加なので呼び出し側変更不要

## Risks

- **`syncStateForwardTask` の lock 取得頻度**: 既存 task の中で lock を 1 回追加するだけ。同期状態更新は秒〜30秒に 1 度なので lock 競合無視可
- **テストでのジャンプ再現**: `EventScheduler.handleJump` を直接呼んでテストする (DriftMonitor 経由は不要、PeerClock facade とは独立した actor 単体テスト)
- **既存 schedule 呼び出し箇所**: `Examples/PeerClockDemo` と `Tests/PeerClockTests/EventSchedulerIntegrationTests.swift` 等で grep 必須
- **`SyncQuality.confidence` の意味論**: 0.0..1.0 の範囲。`NTPSyncEngine` が `min(1.0, filtered.count / syncMeasurements)` で算出している既存ロジック。Phase 3.6 では既存意味論を維持
