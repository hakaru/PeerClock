# PeerClock v0.5 — Public API Surface Cleanup Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Tighten PeerClock's public API surface. Currently (v0.4.2) many implementation-detail actors (`HeartbeatMonitor`, `EventScheduler`, `StatusRegistry`, `StatusReceiver`, `Sleeper` family) are `public` because their nested types or protocols are returned from public `PeerClock` properties. Demote those actors to `internal` by introducing plain-value public wrapper types (`HeartbeatEvent`, etc.) for anything that needs to cross the facade boundary.

**Why v0.5 (breaking):** removing `public` is a source-breaking change for any consumer that imported those types directly. Consumers outside the facade are not expected, but `from: "0.4.x"` SPM pins would silently resolve to v0.5 and fail at build if they did. SemVer says: use a minor bump that clearly indicates breakage. Since this repo is pre-1.0 and we've already broken API between 0.3.x and 0.4.x, a `0.5.0` bump is the right home.

**Architecture:**
- Every type used in a public `PeerClock` signature stays public — this is the minimum viable public API surface.
- Implementation actors (`HeartbeatMonitor`, etc.) become `internal`. Anything they currently expose that crosses the facade boundary is republished as a plain `public struct/enum`.
- `ScheduledEventHandle` keeps a reference to `EventScheduler`; either hide the reference (internal) or wrap the scheduler handle.
- Known v0.4 debts that belong in v0.5 land in this release too: `knownPeers` migration on hot-swap, `star.start()` timeout.

**Tech Stack:** Swift 6.0 strict concurrency. No new external deps.

**Branch (target):** `feat/v0.5.0-api-cleanup`. Base: `v0.4.2`.

**Out of scope (explicitly deferred):**
- Actor-ify `PeerClock` (currently `final class @unchecked Sendable`) — that's a v0.6 design question.
- Push services (NTPSync / CommandRouter / heartbeat / status / drift) down into runtimes — huge refactor; left as a future "v1.0 prep" step.
- Reverse topology transition (star → mesh) — still deferred.
- Dual-stack mesh+star — still deferred.

## Phase 0 — Design decisions to confirm (manual, user-gated)

Before executing, confirm:

1. **Scope of public surface.** Target public list (proposed):
   - Core: `PeerClock`, `Configuration`, `PeerClockError`, `PeerID`
   - Peer model: `Peer`, `PeerStatus`, `DeviceInfo`, `Platform`, `ConnectionState`, `RemotePeerStatus`
   - Topology: `Topology`, `StarRole`, `AutoHeuristic`
   - Commands: `Command`
   - Sync: `SyncState`, `SyncSnapshot`, `SyncQuality`
   - Scheduling: `ScheduledEventHandle`, `ScheduledEventState`, `SchedulerEvent`
   - Observability: `ConnectionEvent`, `HeartbeatEvent` (new wrapper)
   - Convenience: `StatusKeys`, `StatusValueEncoder` (may be optional — ask if any app relies)
   - Everything else: `internal`

2. **`HeartbeatMonitor.Event` handling.** Currently returned from `PeerClock.heartbeatEvents: AsyncStream<HeartbeatMonitor.Event>`. Demoting `HeartbeatMonitor` to `internal` requires either:
   - **(a)** New public `HeartbeatEvent` struct mirroring the fields (peerID, state). Rename `heartbeatEvents` return type. **Source-breaking.** *Recommended.*
   - **(b)** Keep `HeartbeatMonitor.Event` public but make `HeartbeatMonitor` itself internal — **not allowed by Swift** (nested type can't be more accessible than outer).

3. **`EventScheduler` handling.** `ScheduledEventHandle` (public) holds a reference to `EventScheduler`. Options:
   - **(a)** Make `EventScheduler` internal; `ScheduledEventHandle` stores it via an opaque wrapper. Public API unchanged for consumers.
   - **(b)** Keep `EventScheduler` public — this hides the real problem.
   - **(a) recommended.**

4. **`Sleeper` / `RealSleeper` / `MockSleeper`.** These are time-injection abstractions used by `EventScheduler`. Purely testing. Should be `internal` (they're never called from outside PeerClock). **No wrapper needed.**

5. **`StatusKeys` / `StatusValueEncoder`.** These are convenience constants + utilities for apps building status entries. If any PeerClockApps app uses them, keep public. If not, make internal.

6. **`knownPeers` migration on hot-swap.** Gemini's v0.4.1 review question. When `.auto` transitions mesh→star, `restartServices` resets `knownPeers = [localPeerID]`, discarding the previously-discovered peer set. Should the new star runtime inherit? Design options:
   - **(a)** Migrate: `knownPeers` survives hot-swap.
   - **(b)** Reset: current behavior; simpler but might cause brief "no peers" flicker on transition.
   - *Proposal:* **(a)** — migration is cleaner UX.

7. **`star.start()` hang timeout.** If `StarRuntime.start()` hangs (Bonjour permission denied, no listener bind, etc.), `AutoRuntime.performTransition` awaits indefinitely, and `PeerClock.performMeshToStarTransition` with it.
   - **(a)** Add a timeout parameter (default ~10s) — wraps `star.start()` in `Task` + `withTimeout`.
   - **(b)** Rely on user's `PeerClock.stop()` to break out via the new isStopped guard.
   - *Proposal:* **(a)** — defensive; users shouldn't have to think about timeouts for star startup.

import Foundation

/// Observability event for a peer's heartbeat-derived connection state transition.
///
/// Replaces the v0.4 `HeartbeatMonitor.Event` (which was accidentally public
/// because its containing actor was). v0.5 demotes the internals and hands
/// consumers this plain-value type instead.
public struct HeartbeatEvent: Sendable, Equatable {
    public let peerID: PeerID
    public let state: ConnectionState
    public let at: Date

    public init(peerID: PeerID, state: ConnectionState, at: Date = Date()) {
        self.peerID = peerID
        self.state = state
        self.at = at
    }
}

```swift
internal extension HeartbeatMonitor.Event {
    var asPublicEvent: HeartbeatEvent {
        HeartbeatEvent(peerID: peerID, state: state)
    }
}
```

- [ ] **Step 3: Rewire `PeerClock.heartbeatEvents`**

```swift
public var heartbeatEvents: AsyncStream<HeartbeatEvent> {
    let monitorStream = lock.withLock { heartbeatMonitor }?.events
        ?? AsyncStream { $0.finish() }
    return AsyncStream { cont in
        Task {
            for await event in monitorStream {
                cont.yield(event.asPublicEvent)
            }
            cont.finish()
        }
    }
}
```

**Intent:** When `.auto` hot-swaps mesh→star, don't reset `knownPeers = [localPeerID]` — carry the previously-discovered set across the swap so consumers of `PeerClock.peers` don't observe a momentary collapse to "just me".

### Task 3.2: `star.start()` timeout

**Files:**
- Modify: `Sources/PeerClock/Facade/AutoRuntime.swift` — add `startTimeout: Duration = .seconds(10)` parameter
- Modify: `Sources/PeerClock/PeerClock.swift` — `handleTransition` wraps with timeout

**Intent:** If `StarRuntime.start()` hangs (Bonjour permission denied, port bind failure, etc.), `AutoRuntime.performTransition` currently awaits indefinitely. Wrap with a timeout and bail back to mesh on expiry.

- [ ] **Step 1: Add timeout helper**

In `AutoRuntime`, wrap the `star.start()` call:

```swift
let timeoutNs = UInt64(startTimeout.components.seconds) * 1_000_000_000
let started = await withTaskGroup(of: Bool.self, returning: Bool.self) { group in
    group.addTask {
        do { try await star.start(); return true } catch { return false }
    }
    group.addTask {
        try? await Task.sleep(nanoseconds: timeoutNs)
        return false
    }
    let result = await group.next()!
    group.cancelAll()
    return result
}
guard started else {
    // star.start() timed out or threw. Leave mesh active.
    throw AutoRuntimeError.starStartTimeout
}
```

Add a new `AutoRuntimeError` internal enum for this.

- [ ] **Step 2: Test**

Fault-inject a blocking StarRuntime. Assert performTransition throws after ~timeout.

## Risk

1. **Hidden consumers of demoted types.** The only known consumer is `PeerClockApps`, which only uses the `PeerClock` facade — no risk there. External consumers (if any) would see breakage on upgrade. CHANGELOG lists all demotions clearly.
2. **`HeartbeatEvent` translation overhead.** The internal-to-public translation spawns a `Task` per subscribe. Acceptable — heartbeat events are low-rate (≤1 Hz). If overhead concerns arise, switch to an `AsyncStream.makeStream` + subscriber pattern.
3. **`knownPeers` migration semantics.** If a peer was in `knownPeers` on mesh but isn't on star (because star's transport won't see it), consumers may briefly observe a stale peer. Acceptable for ~3s until star re-discovers; document.
4. **`star.start()` timeout value.** 10s default may be too short for first-run Bonjour permission dialogs. Reconfirm on real devices; may need 30s default with an option.
