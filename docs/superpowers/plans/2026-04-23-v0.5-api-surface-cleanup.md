# PeerClock v0.5 — Public API Surface Cleanup Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal (v2, after Codex + Gemini independent review):** Demote a **specific, bounded set** of implementation-detail types to `internal`, introduce a public wrapper where needed (`HeartbeatEvent`), absorb two v0.4 debts (`knownPeers` continuity, `star.start()` timeout), and **fully remove** the deprecated `send(_:to:)` that has been `@available(deprecated)` since v0.4.0.

**Explicitly NOT a full public-surface audit.** The repo currently exposes ~68 public types (grep `^public` under `Sources/PeerClock/`). This plan touches only the types the v0.4 review flagged as obvious leaks; deciding per-type whether Transport / WiFiTransport / StarTransport / MessageCodec / HostElection / BonjourAdvertiser / etc. should be `internal` is deferred to **v0.6 per-category audit** (a separate plan). v0.5's freeze list below is **additive** — "these types are locked as public for v0.5 callers"; it does **not** claim "everything else becomes internal".

**Why v0.5 (breaking):** dropping `public` on the 5 actors + changing `heartbeatEvents` return type + removing `send(_:to:)` are all source-breaking for anyone who directly referenced those symbols. SemVer says: minor bump with a migration guide. Pre-1.0 so the bump is acceptable.

**Architecture:**
- Types used in a public `PeerClock` signature stay public (foundational — see freeze list).
- 5 implementation actors (`HeartbeatMonitor`, `EventScheduler`, `StatusRegistry`, `StatusReceiver`, `Sleeper` family) become `internal`. `HeartbeatEvent` (new) wraps the formerly-leaked nested `HeartbeatMonitor.Event` for public consumption — strict mirror of today's data (`peerID` + `state`), no invented fields.
- `ScheduledEventHandle` stops publicly exposing `EventScheduler`.
- v0.4 debts folded in: `knownPeers` continuity across hot-swap (fix extends deeper than restartServices — see Task 3.1); `star.start()` timeout as a `Configuration.starStartTimeout` knob, default **30s** (realistic for first-run Bonjour permission prompts; Codex/Gemini review flagged 10s as too aggressive). Nil permitted for no-timeout.
- `send(_:to:)` deleted outright (deprecated since v0.4.0 — Codex/Gemini both recommended full removal in a breaking release).

**Tech Stack:** Swift 6.0 strict concurrency. No new external deps.

**Branch (target):** `feat/v0.5.0-api-cleanup`. Base: `v0.4.2`.

**Out of scope (explicitly deferred):**
- **Broader public-surface audit** (remaining ~40 public types under Transport / Star / Wire / Election / Discovery / etc.) → **v0.6 per-category plan**.
- Actor-ify `PeerClock` → v0.6 design question.
- Push services (NTPSync / CommandRouter / heartbeat / status / drift) into runtimes → v1.0 prep.
- Reverse topology transition (star → mesh) → still deferred.
- Dual-stack mesh+star → still deferred.

## Phase 0 — Design decisions to confirm (manual, user-gated)

Before executing, confirm:

1. **Scope of this v0.5 release — 5 actors + 1 protocol family.** The **only** types v0.5 demotes are:
   - `HeartbeatMonitor` (actor) + its nested `Event`
   - `EventScheduler` (actor)
   - `StatusRegistry` (actor)
   - `StatusReceiver` (actor)
   - `Sleeper` (protocol), `RealSleeper` (struct), `MockSleeper` (actor)
   
   Everything else stays at its current visibility. The broader per-category audit (Transport / Star internals / Wire / Election / Discovery) is deferred to v0.6. This is the "promise shrinkage" adjustment from the Codex review.

2. **`HeartbeatMonitor.Event` handling.** Currently returned from `PeerClock.heartbeatEvents: AsyncStream<HeartbeatMonitor.Event>`. Demoting `HeartbeatMonitor` to `internal` requires either:
   - **(a)** New public `HeartbeatEvent` struct mirroring the fields (peerID, state). Rename `heartbeatEvents` return type. **Source-breaking.** *Recommended.*
   - **(b)** Keep `HeartbeatMonitor.Event` public but make `HeartbeatMonitor` itself internal — **not allowed by Swift** (nested type can't be more accessible than outer).

3. **`EventScheduler` handling.** `ScheduledEventHandle` (public) holds a reference to `EventScheduler`. Options:
   - **(a)** Make `EventScheduler` internal; `ScheduledEventHandle` stores it via an opaque wrapper. Public API unchanged for consumers.
   - **(b)** Keep `EventScheduler` public — this hides the real problem.
   - **(a) recommended.**

4. **`Sleeper` / `RealSleeper` / `MockSleeper`.** These are time-injection abstractions used by `EventScheduler`. Purely testing. Should be `internal` (they're never called from outside PeerClock). **No wrapper needed.**

5. **`StatusKeys` / `StatusValueEncoder`.** These are convenience constants + utilities for apps building status entries. If any PeerClockApps app uses them, keep public. If not, make internal.

6. **`knownPeers` migration on hot-swap.** Gemini's v0.4.1 review question. When `.auto` transitions mesh→star, `restartServices` resets `knownPeers = [localPeerID]`, discarding the previously-discovered peer set. Should the new star runtime inherit? Design options:
   - **(a)** Migrate: `knownPeers` survives hot-swap.
   - **(b)** Reset: current behavior; simpler but might cause brief "no peers" flicker on transition.
   - *Proposal:* **(a)** — migration is cleaner UX.

7. **`star.start()` hang timeout.** If `StarRuntime.start()` hangs (Bonjour permission denied, no listener bind, etc.), `AutoRuntime.performTransition` awaits indefinitely, and `PeerClock.performMeshToStarTransition` with it.
   - **(a)** Add a timeout parameter (default ~10s) — wraps `star.start()` in `Task` + `withTimeout`.
   - **(b)** Rely on user's `PeerClock.stop()` to break out via the new isStopped guard.
   - *Proposal:* **(a)** — defensive; users shouldn't have to think about timeouts for star startup.

- [ ] **Step 1: New `HeartbeatEvent` struct — strict mirror of source data**

Current `HeartbeatMonitor.Event` (HeartbeatMonitor.swift:9-16) has **only** `peerID: PeerID` and `state: ConnectionState`. No timestamp. The public wrapper must match — do **not** invent a `Date` field (Codex review flagged an earlier draft for a dead `at` field).

```swift
import Foundation

/// Observability event for a peer's heartbeat-derived connection state transition.
///
/// Replaces v0.4's `HeartbeatMonitor.Event` (accidentally public because its
/// outer actor was). v0.5 demotes the internals and hands consumers this
/// plain-value type instead. Fields mirror the internal source 1:1 — no
/// timestamp is emitted (adding one would require source-side changes and
/// is deferred).
public struct HeartbeatEvent: Sendable, Equatable {
    public let peerID: PeerID
    public let state: ConnectionState

    public init(peerID: PeerID, state: ConnectionState) {
        self.peerID = peerID
        self.state = state
    }
}
```

- [ ] **Step 2: Demote `HeartbeatMonitor`**

Change `public actor HeartbeatMonitor` → `internal actor HeartbeatMonitor`. Its nested `Event` becomes internal too. Add an internal extension translating to the public `HeartbeatEvent`:

```swift
internal extension HeartbeatMonitor.Event {
    var asPublicEvent: HeartbeatEvent {
        HeartbeatEvent(peerID: peerID, state: state)

}
```

- [ ] **Step 3: Rewire `PeerClock.heartbeatEvents` — single-task forwarder pattern**

**Critical correction from Codex/Gemini review:** do NOT spawn a Task per subscribe. The existing `connectionEvents` wiring (PeerClock.swift:154-156, `connectionEventsForwardTask` spawned inside `restartServices`) already forwards events from `runtime.connectionEvents` to a facade continuation via **one** task per service lifecycle. `heartbeatEvents` should follow the same pattern.

Implementation:
1. Convert `heartbeatEvents` from a computed property to a **stored** `AsyncStream<HeartbeatEvent>` (with a matching `heartbeatEventsContinuation` stored continuation), mirroring how `connectionEvents` is stored (PeerClock.swift:42, 155).
2. In `restartServices`, after the heartbeat monitor is created, spawn ONE forwarder task (parallel to the `eventsForwardTask` already there for connectionEvents):
   ```swift
   let heartbeatEventsCont = self.heartbeatEventsContinuation
   let hbEventsForwardTask: Task<Void, Never>? = heartbeat.map { hb in
       Task {
           for await event in hb.events {
               heartbeatEventsCont.yield(event.asPublicEvent)
           }
       }
   }
   // Store under lock alongside the other routing tasks.
   ```
3. Track the task in `teardownServices` (add to the cancel-await parallel fan-out, same as `eventsForwardTask`).
4. Public accessor becomes a simple `public let heartbeatEvents: AsyncStream<HeartbeatEvent>` — no per-access cost.

## Phase 3 — v0.4 debt cleanup

### Task 3.1: `knownPeers` continuity across hot-swap

**Critical correction from Codex review:** the earlier single-line fix to `restartServices` is insufficient. `runCoordinationLoop` at PeerClock.swift:763-767 does `knownPeers = peers` unconditionally on every iteration — so the first (empty) peer snapshot from the new star runtime still collapses `pc.peers` for subscribers. The fix has to cover both sites.

**Files:**
- Modify: `Sources/PeerClock/PeerClock.swift` — both `restartServices(transport:)` AND `runCoordinationLoop(peerStream:)`

- [ ] **Step 1: Preserve prior set on restartServices**

```swift
// was:
knownPeers = [localPeerID]

// becomes (inside the lock in restartServices):
// Preserve prior peer set on hot-swap. On a fresh start() knownPeers is empty,
// so this still seeds with localPeerID.
if knownPeers.isEmpty { knownPeers = [localPeerID] }
```

- [ ] **Step 2: Merge rather than overwrite in runCoordinationLoop**

Add an internal `pendingTransitionalMerge: Bool` flag (under lock) set by `handleTransition` after a successful swap and consumed by the first `runCoordinationLoop` iteration. When the flag is set, the new empty snapshot is **merged** into `knownPeers` (union instead of overwrite) and the flag cleared. Subsequent iterations use the normal overwrite semantics.

```swift
// Roughly, in runCoordinationLoop's inner lock.withLock:
let prev = knownPeers
let merged: Set<PeerID>
if consumeTransitionalMerge() {
    merged = prev.union(peers)   // keep prior peers visible during bootstrap
} else {
    merged = peers                // normal steady-state
}
knownPeers = merged
```

`consumeTransitionalMerge()` is a tiny private method that atomically reads+clears the flag.

Document in ARCHITECTURE / spec that peers from the prior topology may remain visible for up to `heartbeatDisconnectedAfter` (~3-5s) after the transition — they drop naturally when the star runtime's heartbeat stops seeing them.

- [ ] **Step 1: Add `Configuration.starStartTimeout`**

```swift
/// Timeout applied around `StarRuntime.start()` during an `.auto`
/// mesh→star transition. Defaults to 30s — covers the first-run
/// Local Network permission dialog on iOS/macOS. Set to `nil` to
```

- [ ] **Step 2: AutoRuntime accepts + honors the timeout**

```swift
// AutoRuntime.init gains: startTimeout: Duration? = .seconds(30)
// In performTransition, wrap star.start() using a TaskGroup race:

guard let timeout = startTimeout else {
    try await star.start()  // unbounded
    // fall through to success path
    return
}

let timeoutNs = Self.nanoseconds(from: timeout)
let started = await withTaskGroup(of: Result<Void, Error>.self) { group in
    group.addTask {
        do { try await star.start(); return .success(()) } catch { return .failure(error) }
    }
    group.addTask {
        try? await Task.sleep(nanoseconds: timeoutNs)
        return .failure(AutoRuntimeError.starStartTimeout)
    }
    defer { group.cancelAll() }
    return await group.next() ?? .failure(AutoRuntimeError.starStartTimeout)
}

switch started {
case .success: break
case .failure(let err): throw err
}
```

Add `internal enum AutoRuntimeError: Error { case starStartTimeout }`.

## Risk

1. **Hidden consumers of demoted types.** The only known consumer is `PeerClockApps`, which only uses the `PeerClock` facade — no risk there. External consumers (if any) would see breakage on upgrade. CHANGELOG lists all demotions clearly.
2. **`HeartbeatEvent` translation overhead.** The internal-to-public translation spawns a `Task` per subscribe. Acceptable — heartbeat events are low-rate (≤1 Hz). If overhead concerns arise, switch to an `AsyncStream.makeStream` + subscriber pattern.
3. **`knownPeers` migration semantics.** If a peer was in `knownPeers` on mesh but isn't on star (because star's transport won't see it), consumers may briefly observe a stale peer. Acceptable for ~3s until star re-discovers; document.
4. **`star.start()` timeout value.** 10s default may be too short for first-run Bonjour permission dialogs. Reconfirm on real devices; may need 30s default with an option.
